# 涂装生产排程管理系统技术与功能模块说明

## 1. 文档目的

这份文档用于从整体上介绍当前项目的业务目标、功能模块、技术结构和核心实现思路。

## 2. 项目定位

本项目是一个基于 Django 的涂装生产排程管理系统，核心目标是围绕“涂装双层滚动排产”建立一套完整的业务闭环。

它解决的不是单一的排产计算问题，而是以下几个连续问题：

1. 如何维护排产所依赖的基础数据。
2. 如何导入库存、总成拉动、安全库存等生产数据。
3. 如何基于规则生成短期和长期排产计划。
4. 如何把排产结果以阵型、库存更新、历史记录等形式保存下来。
5. 如何让用户持续追踪风险、查看历史、导出结果和接收提醒。

因此，这个项目不是“一个算法脚本”，而是“一个围绕排产算法构建的业务系统”。

## 3. 系统目标与业务场景

系统服务的核心场景可以概括为：

- 维护车型、颜色、产品、系统参数等基础配置。
- 导入涂装库存、注塑库存、安全库存、总成拉动数据。
- 基于短期窗口和长期窗口计算风险并生成排产计划。
- 根据排产结果生成阵型槽位、库存快照和历史记录。
- 在结果页、历史页和导出文件中对计划进行追溯和解释。

从业务角度看，这个系统关注的是“滚动排产”：

- 读取一段短期拉动数据，识别临近断供风险。
- 再读取一段长期拉动数据，识别中期风险组。
- 将短期与长期计划整合为一圈阵型。
- 根据最终阵型更新涂装库存和注塑库存。
- 将本次更新后的库存作为后续计算的实际库存基础。

## 4. 整体功能模块

项目目前可以分为五个核心功能域。

### 4.1 账户与访问控制

对应模块：`accounts`

主要职责：

- 用户登录和登出。
- 登录态控制。
- 为系统首页和各业务页面提供访问入口。

这一层相对轻量，但它决定了系统是一个可交互的 Web 应用，而不是离线工具。

### 4.2 基础数据与生产数据管理

对应模块：`data`

主要职责：

- 维护车型、颜色、位置类型、产品。
- 维护涂装库存、注塑库存、安全库存。
- 维护总成拉动数据。
- 维护系统参数。
- 支持数据的增删改查、筛选、分页、批量删除、导出和导入。

这是整个系统的数据底座。没有这一层，排产模块就没有输入来源。

### 4.3 排产计算与结果追踪

对应模块：`schedule`

主要职责：

- 计算短期需求、长期需求。
- 计算短期风险、长期风险。
- 生成短期计划和长期计划。
- 形成阵型槽位。
- 更新库存快照。
- 保存排产记录、需求记录、风险记录、计划记录、阵型槽位和库存快照。
- 提供结果详情页、历史页、Excel 导出。

这一层是系统的核心业务模块，也是系统价值最集中的部分。

### 4.4 风险提醒与消息通知

对应模块：`notifications`

主要职责：

- 在排产发现风险时生成通知。
- 将短期风险、长期风险和相关排产记录与用户关联。
- 提供消息查看入口。

这一层把“计算结果”转换成“可感知的提醒”，提升系统的业务可用性。

### 4.5 项目级基础设施

对应模块：`painting`

主要职责：

- Django 项目的全局配置。
- 主路由分发。
- 模板上下文。
- 中间件控制。

例如当前项目增加了未迁移数据库时的拦截机制，避免系统在数据库结构不一致时直接报错。

## 5. 项目目录结构说明

结合当前代码结构，可以从目录层面理解项目：

```text
painting/
├─ accounts/         用户登录登出
├─ data/             基础配置、库存、总成拉动、导入导出
├─ notifications/    风险通知
├─ schedule/         排产算法、结果保存、历史与导出
├─ painting/         全局配置、路由、中间件
├─ templates/        通用模板和各模块页面
├─ static/           静态资源
├─ media/            导入临时文件和媒体目录
├─ doc/              业务规则文档
├─ docs/             设计和实现记录
└─ db.sqlite3        默认 SQLite 数据库
```

从这个结构可以看出，本项目采用的是较典型的 Django 单体应用方式：每个业务域作为一个 app，统一由项目层进行路由和配置管理。

## 6. 关键数据模型

系统的数据模型可以分为两层：生产基础数据模型 和 排产过程数据模型。

### 6.1 生产基础数据模型

位于 `data/models.py`。

主要包括：

- `VehicleModel`
  - 表示车型。
- `Color`
  - 表示颜色，并带有英文颜色到中文展示名和色值的映射逻辑。
- `PositionType`
  - 表示前件、后件。
- `Product`
  - 车型 + 颜色 + 位置的组合，是排产、库存和风险计算的核心对象。
- `Inventory`
  - 涂装库存。
- `InjectionInventory`
  - 注塑库存。
- `SafetyStock`
  - 安全库存。
- `AssemblyPullData`
  - 总成拉动顺序数据。
- `SystemParameter`
  - 系统参数，如一圈车数、短期/长期产能百分比、前后平衡约束等。
- `ImportRecord`
  - 导入归档记录，用于记录导入文件、状态、成功/失败条数和错误明细。

这些模型共同构成排产算法的输入层。

### 6.2 排产过程数据模型

位于 `schedule/models.py`。

主要包括：

- `ScheduleRecord`
  - 一次完整排产计算的主记录。
- `DemandRecord`
  - 每次排产中短期或长期的需求记录。
- `RiskRecord`
  - 每次排产中短期或长期的风险记录。
- `SchedulePlan`
  - 每次排产中生成的短期/长期计划。
- `FormationSlot`
  - 最终阵型槽位。
- `InventorySnapshot`
  - 本次计算对应的库存快照。

这一层体现了本项目的重要设计思想：

- 排产不是一次“即时计算后即丢弃”的过程。
- 排产是可追溯、可复盘、可导出的业务事件。

## 7. 核心业务流程

系统主流程可以概括为以下链路：

1. 维护基础配置。
2. 导入生产数据。
3. 发起排产计算。
4. 保存需求、风险、计划、阵型和库存快照。
5. 生成结果页面、历史记录和导出文件。
6. 根据风险生成通知。

如果展开来看，可以分为以下步骤。

### 7.1 数据准备

用户首先需要维护：

- 车型
- 颜色
- 产品
- 系统参数

然后导入或录入：

- 涂装库存
- 注塑库存
- 安全库存
- 总成拉动数据

这些数据完成后，排产模块才具备运行条件。

### 7.2 发起排产

用户在排产计算页面设置短期时长和长期时长，系统会：

- 读取系统参数。
- 读取库存和安全库存。
- 读取总成拉动数据。
- 读取上一轮已完成排产的阵型，用于阵型复用参考。

### 7.3 生成需求与风险

算法会按窗口读取总成拉动数据：

- 短期窗口：前一段拉动记录。
- 长期窗口：紧接短期窗口之后的拉动记录。

然后按产品计算：

- 需求数量
- 折算生产数量
- 短期终值风险
- 长期风险值
- 长期组风险值

### 7.4 生成计划与阵型

当前实现中，排产计划主要分为：

- 短期计划
  - 优先解决短期缺货项。
- 长期计划
  - 基于长期风险组和约束进行分配。

之后再合并为阵型槽位：

- 尽量复用上一轮槽位中的同类产品。
- 对剩余计划进行顺序排布。

### 7.5 更新库存并归档

排产结果会生成库存快照，并将：

- 涂装更新后库存
- 注塑更新后库存

写回库存表中，供后续排产继续使用。

同时保存：

- 本次排产记录
- 本次需求、风险、计划、阵型、库存快照

这保证了历史记录和结果详情都可以追踪。

## 8. 排产模块的技术实现思路

排产逻辑主要位于 `schedule/services/algorithms.py`。

从实现方式上看，这个模块采用了“服务类 + 数据持久化”的模式：

- `SchedulingAlgorithm`
  - 负责参数读取、库存读取、需求计算、风险计算、计划生成、阵型构造和库存更新。
- `save_results`
  - 负责将内存中的计算结果保存为数据库记录。

### 8.1 为什么采用服务类

因为排产是一个跨多个模型、多步骤的连续过程。

如果把逻辑直接散落在视图中，会导致：

- 难以测试
- 难以复用
- 难以定位算法问题

把它集中在服务类中，更方便：

- 单独调试算法
- 编写测试
- 逐步增强规则

### 8.2 当前实现的几个关键点

1. 参数驱动
   - 一圈车数、短期/长期产能百分比、前后平衡差值等都来自 `SystemParameter`。
2. 双库存体系
   - 同时读取涂装库存和注塑库存，分别参与不同阶段的判断。
3. 排产过程持久化
   - 需求、风险、计划、槽位、库存快照都会落库。
4. 历史可追踪
   - 结果页、历史页、导出文件共享同一批排产记录。

### 8.3 当前实现仍需持续优化的方向

从项目现状看，排产模块仍处于持续迭代中，后续常见优化方向包括：

- 强化“满圈生产”的口径。
- 强化阵型不变约束。
- 优化长期计划与实际总车数的关系。
- 让计划说明和结果页解释更可读。

也就是说，算法模块已经形成了骨架，但仍然是项目中最需要持续演进的部分。

## 9. 数据管理模块说明

`data` 模块是系统中工作量非常大的一层，它不仅承担数据维护，还承担导入、导出、筛选和历史归档。

### 9.1 页面类型

当前页面大致包括：

- 基础配置页面
  - 车型
  - 颜色
  - 产品
  - 系统参数
- 生产数据页面
  - 涂装库存
  - 注塑库存
  - 安全库存
  - 总成拉动
- 导入页面
  - 模板下载
  - 文件导入
  - 错误明细
  - 导入历史

### 9.2 当前实现特点

- 基本具备前台 CRUD。
- 支持筛选、分页、批量删除。
- 支持按当前筛选结果导出 CSV。
- 支持 Excel 导入。
- 支持导入记录归档和错误明细追踪。

这意味着 `data` 模块不仅是后台配置界面，也是生产数据运维工具。

## 10. 历史记录与结果展示模块

排产结果不是只展示一张“成功/失败”的页面，而是形成了三种输出载体：

### 10.1 结果详情页

用于查看单次计算的详细内容，包括：

- 需求
- 风险
- 计划
- 阵型槽位
- 库存更新
- 注塑约束
- 计划缺口

### 10.2 历史页

用于查看多次排产的总体情况，包括：

- 时间
- 状态
- 短期/长期窗口
- 复用槽位
- 总成拉动摘要
- 库存更新摘要
- 注塑约束摘要
- 删除历史记录

### 10.3 Excel 导出

用于离线分析和存档，通常会包含：

- 计划表
- 阵型表
- 库存更新表
- 缺口说明
- 约束说明

这三者共同构成了“计算结果可解释、可复盘、可留档”的输出体系。

## 11. 通知模块说明

通知模块位于 `notifications` app。

当排产模块检测到明显的短期风险或长期风险时，会创建通知记录，并关联到相关排产记录。

它的作用不是替代结果页，而是让用户在首页或通知入口可以更快感知问题。

这个设计说明项目不仅关注“算出来”，也关注“让用户知道出问题了”。

## 12. 技术架构概览

从技术栈上看，当前项目属于典型的 Django 服务端渲染应用：

- 后端框架：Django
- 数据库：SQLite
- 前端形态：Django Template + Bootstrap/Tabler
- 静态资源：`static/`
- 媒体文件：`media/`
- 环境配置：`.env`

### 12.1 当前架构特点

- 开发和部署门槛较低。
- 页面开发效率较高。
- 适合中小型业务系统快速迭代。
- 业务逻辑集中在 Python 服务层，便于理解和调试。

### 12.2 适合的项目阶段

当前架构非常适合：

- 需求变化还比较频繁的阶段。
- 业务规则仍在不断收敛的阶段。
- 以内部系统、生产管理系统为主的场景。

## 13. 系统运行方式

从运行方式上看，当前项目是标准的 Django 项目：

1. 安装依赖。
2. 配置 `.env`。
3. 执行数据库迁移。
4. 启动开发服务器。

典型命令：

```bash
python manage.py migrate
python manage.py runserver
```

如果数据库结构与代码不一致，项目中间件会优先拦截请求并提示先执行迁移，这一点提升了系统的可维护性。

## 14. 适合团队理解这个项目的方式

如果是新同事接手项目，建议按以下顺序理解：

1. 先看 `data/models.py`
   - 先理解基础数据是什么。
2. 再看 `schedule/models.py`
   - 理解排产结果是如何持久化的。
3. 再看 `schedule/services/algorithms.py`
   - 理解排产核心逻辑。
4. 最后看 `data/views.py`、`schedule/views.py`
   - 理解页面与业务流程如何衔接。

这样比直接从模板或零散页面入口开始更高效。

## 15. 总结

从整体上看，这个项目已经具备一个完整排产系统的核心骨架：

- 有基础配置层
- 有生产数据层
- 有排产算法层
- 有结果追踪层
- 有通知提醒层

它的价值不只在“排产计算”，而在于把排产前的数据准备、排产中的规则计算、排产后的追踪与导出统一到了一个系统里。

如果用一句话概括这个项目，可以这样描述：

> 这是一个以 Django 为基础、以涂装双层滚动排产为核心、覆盖数据管理、排产计算、历史追踪和通知提醒的业务管理系统。

后续如果继续完善，这个项目最值得持续投入的方向仍然是：

- 排产算法的业务一致性
- 阵型与产能口径的统一
- 结果解释能力
- 历史与导出的可追溯性

但从当前结构来看，项目已经具备较好的扩展基础。
